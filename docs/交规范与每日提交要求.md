# 提交规范与每日提交要求

本项目按“过程管控”验收要求，采用小粒度、每日提交、备注清晰的策略。

## 1. 提交频率

- 每日最少 1 次提交
- 每个功能点拆为 1~3 个提交，不混合无关改动

## 2. 提交粒度

单次提交建议只做一类改动：

- `feat`：新功能（例如新增规则接口）
- `fix`：缺陷修复（例如幂等重复发布）
- `refactor`：重构（不改行为）
- `docs`：文档
- `test`：测试

## 3. 提交信息模板

优先使用验收更直观的中文模块格式：

```text
【运单模块】新增载重体积双校验
【MQ】实现死信队列转发
【结算模块】修复规则删除假成功
```

如团队内部继续沿用 conventional commit，也必须至少满足“模块 + 动作 + 目的”三要素，例如：

```text
feat(waybill): add weight-volume dual validation
fix(mq): prevent duplicate consumer replay on dead-letter retry
```

禁止使用以下备注：

- `fix`
- `修改`
- `xu-wuliu`
- `更新代码`

推荐正文模板：

```text
【模块】一句话说明本次改动

背景:
- <为什么改>

改动:
- <关键改动1>
- <关键改动2>

验证:
- <命令和结果>
```

示例：

```text
【结算模块】新增定价规则管理接口

背景:
- 支持结算规则配置化，快速微调

改动:
- 新增 GET/POST /api/pricing-rules
- 新增 POST /api/pricing-rules/reload
- 启动时从 pricing_rule 表加载规则

验证:
- npm run check --workspace @waybill/api 通过
- 手工调用接口返回 200/201
```

本仓库已提供提交模板文件：`.gitmessage.txt`

本机建议执行：

```text
git config commit.template .gitmessage.txt
```

## 4. 每日提交清单（建议）

- 业务功能提交
- 文档同步提交
- 验证/测试提交

## 5. 备注清晰要求

提交说明必须包含：

- 改动目标（解决什么问题）
- 关键逻辑（做了什么）
- 验证结论（如何证明）

## 6. 自动校验

仓库已提供历史提交校验脚本：

```text
npm run check:git-history
```

校验内容：

- 是否存在每日提交记录
- 是否存在泛化备注（如 `fix` / `修改`）
- 是否存在一次提交修改过多文件（阈值默认 30 个）
