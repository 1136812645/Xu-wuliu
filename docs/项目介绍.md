# 项目清单与项目介绍

## 1. PDF 需求分析

根据 PDF 内容，这个项目不是普通的运单录入系统，而是一套面向物流企业内部管理的运单与结算管理后台，目标是同时覆盖业务闭环、分布式架构能力、性能稳定性和完整交付材料。

需求可归纳为 4 个层面：

### 1.1 业务层

- 基础档案管理：货主、承运商、车辆、司机资料维护
- 运单全流程管理：创建、分配、提货、在途、签收、回单上传
- 结算管理：阶梯运价、附加费、保险费、补贴、扣款
- 预警管理：司机证件、车辆证件临期和过期提醒
- 统计报表：运单量、营收、承运商毛利

### 1.2 技术层

- 前后端分离，符合 MVC 架构
- 运单按分表设计，支持未来扩容
- RabbitMQ 驱动运单状态流转，保证一致性
- Redis 支撑缓存、幂等、分布式锁
- 所有对外操作接口支持幂等
- 支持 Google 登录和角色权限控制

### 1.3 运维交付层

- 支持 Docker 部署
- 支持单机模拟分布式部署
- 支持多服务器扩展部署
- 提供运行日志、MQ 日志、问题排查日志

### 1.4 验收层

- 需要完整设计文档
- 需要可运行代码
- 需要自测用例和测试报告
- 需要部署说明
- 需要支撑并发、批量导入、故障排查和高可用演练

## 2. 项目定位

项目名称建议：内部运单与结算管理后台

一句话介绍：

这是一套面向物流企业内部运营、结算和风控场景的后台系统，用于统一管理运单全流程、费用核算、证件预警、报表统计以及分布式业务支撑能力。

适用对象：

- 物流公司运营人员
- 结算专员
- 承运商协同人员
- 平台管理员

核心价值：

- 让运单从创建到结算形成闭环
- 降低重复操作、重复签收、重复回单带来的脏数据风险
- 让费用规则配置化，支持快速调整
- 为后续分布式部署和系统扩容打基础

## 4. 项目介绍

### 4.1 项目背景

物流企业在内部运营中，常见痛点包括：

- 运单状态流转分散，人工跟踪成本高
- 费用核算规则复杂，容易漏算或错算
- 重复签收、重复回单上传导致脏数据
- 车辆和司机证件过期缺乏统一预警
- 高并发开单时容易出现重复运单或运力冲突
- 随业务增长，单表单机架构难以支撑扩容

因此需要建设一套同时具备业务闭环能力和分布式扩展能力的内部后台系统。

### 4.2 项目目标

- 建立统一的运单全流程管理平台
- 建立可配置的结算规则和费用明细体系
- 建立证件预警和运营报表能力
- 建立面向并发、幂等、扩容的技术底座
- 建立标准化部署、测试和问题复盘体系

### 4.3 项目特色

- 业务闭环完整：从开单到签收再到结算都有清晰流程
- 费用逻辑可解释：每项费用独立可追溯
- 架构可扩展：分表、MQ、缓存、双实例部署齐备
- 运维可落地：支持 Docker、日志审计、故障演练
- 验收可对标：围绕 PDF 考核点逐项设计

### 4.4 预期成果

项目完成后，企业可获得一套用于内部运营管理的后台系统原型或正式系统，具备以下能力：

- 支持多角色协作处理物流运单
- 支持复杂运费与结算规则配置
- 支持证件到期预警与风险提醒
- 支持高并发、幂等、分布式扩展设计
- 支持部署演示、功能验证和验收汇报
