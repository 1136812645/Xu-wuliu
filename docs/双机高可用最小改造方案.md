# 双机高可用最小改造方案（两台服务器可落地）

## 1. 背景与目标

现状问题：

- 服务器 1 承载 gateway/web/api-1。
- 服务器 2 承载 api-2/mysql/redis/rabbitmq。
- 当服务器 2 故障时，服务器 1 的 api-1 因无法连接 mysql/redis/rabbitmq，业务不可用。

改造目标：

- 任意一台服务器故障，另一台仍能继续提供核心业务能力。
- 在仅有两台服务器前提下，优先满足验收连续性要求，再逐步完善一致性与自动故障切换。

## 2. 总体策略（最小改造）

采用“同构双活 + 单点入口 + 业务优先”的最小方案：

- 两台机器都部署完整业务栈：gateway/web/api/mysql/redis/rabbitmq。
- 每台机器上的 api 优先连接本机依赖（127.0.0.1 / 容器内服务名）。
- 网关 upstream 同时指向两台 api，保留失败重试与切换。
- 故障时由网关自动切换到存活节点，确保业务连续。

说明：

- 该方案优点是改造快、验收直观。
- 该方案代价是双机数据一致性需要补充机制（见第 8 章）。

## 3. 分阶段实施

### 阶段 A：可用性先达标（1 天）

目标：停掉任意一台机器，另一台可继续登录和办理业务。

实施：

1. 在服务器 1 新增 mysql/redis/rabbitmq 容器。
2. 在服务器 2 保留 mysql/redis/rabbitmq（已存在）。
3. 调整 api-1 使用本机依赖。
4. 调整 api-2 使用本机依赖。
5. 保持 gateway upstream 指向两台 api（192.168.5.81:3001 和 192.168.5.82:3000）。

验收命令：

- 基线
  - curl -s http://127.0.0.1:18080/api/ha/instance
- 停服务器 2
  - 健康仍 200
  - 登录与 dashboard 正常
- 恢复服务器 2
- 停服务器 1
  - 通过服务器 2 网关入口验证业务正常

本仓库演练证据：

- docs/logs/failover-drill-stageA-2026-07-09.md

### 阶段 B：一致性补强（2-3 天）

目标：降低双活数据偏差风险。

实施建议：

1. MySQL：主从复制（s2 主，s1 从）+ 故障时手工升主。
2. Redis：主从 + Sentinel（3 个 Sentinel，2 台机可部署 3 进程）。
3. RabbitMQ：双节点集群 + 策略镜像（或 quorum queue，接受两节点限制）。
4. API：读取写入策略明确化（优先本地，必要时切主配置）。

### 阶段 C：自动切换（后续）

目标：减少人工干预。

实施建议：

1. 入口层加 Keepalived 或上层 LB 健康探测。
2. 数据层引入自动提升与漂移监控。
3. 增加“切换演练脚本”与“回切脚本”。

## 4. 当前仓库建议改造点

## 4.1 新增部署清单

建议新增：

- deploy/distributed/server1/docker-compose.ha.yml
- deploy/distributed/server2/docker-compose.ha.yml
- deploy/distributed/ha/drill-failover.ps1
- deploy/distributed/ha/drill-recover.ps1

## 4.2 环境变量统一

两台都保留以下变量，并通过 .env 管理：

- DEV_LOGIN_ENABLED
- DEV_LOGIN_PASSWORD
- AUTH_SESSION_SECRET
- GOOGLE_CLIENT_ID
- GOOGLE_ADMIN_EMAILS
- GOOGLE_CARRIER_EMAILS
- GOOGLE_SHIPPER_EMAILS

## 4.3 验收口径脚本化

建议将以下验收操作固化为脚本：

1. 基线健康检查。
2. 停一台服务器后的登录与核心接口验证。
3. 恢复后回归验证。
4. 结果输出到 docs/logs/ 时间戳文件。

## 5. 一次完整故障验收流程（可复制）

1. 基线：

- /health == 200
- /api/ha/instance 可返回 api-1 或 api-2
- 登录成功

2. 故障注入：

- 在目标机执行 docker compose down

3. 连续性验证：

- 再次验证 /health
- 执行登录接口
- 执行 /api/dashboard

4. 恢复：

- docker compose up -d
- 再次验证登录与 dashboard

5. 证据留存：

- 保存命令输出到 docs/logs/failover-日期时间.md

## 6. 验收通过标准（针对“停一台仍可用”）

判定为“通过”需满足：

1. 任意一台停机后，公网入口仍可访问。
2. 登录接口可用（非仅 health 存活）。
3. 至少一个核心业务接口可成功返回（如 dashboard、waybills）。
4. 恢复后系统可回归双节点。

## 7. 风险与注意事项

1. 双活本地依赖模式会带来数据分叉风险。
2. 若验收更关注“业务连续”而非“强一致”，该方案可快速过线。
3. 若后续进生产，必须完成阶段 B/C。

## 8. 建议执行顺序

1. 先完成阶段 A，拿到连续性验收通过。
2. 再做阶段 B，补齐数据一致性。
3. 最后做阶段 C，降低人工操作成本。

## 9. 结论

在仅有两台服务器的前提下，建议先采用“同构双活 + 本地依赖优先”的最小高可用方案完成验收，再逐步升级为具备更强一致性和自动切换能力的生产方案。
