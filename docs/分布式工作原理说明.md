# 分布式工作原理说明

## 1. 说明目的

本文档解释以下两条验收场景背后的系统工作机制：

1. 人为关闭其中一台服务器，另外一台继续工作。
2. 重启一台服务器后，两台同时工作，系统支持分布式工作。

## 2. 系统结构

当前系统采用双节点 API + 网关转发模式：

- 服务器 1：gateway、web、api-1（并具备本机依赖服务能力）
- 服务器 2：api-2（并具备本机依赖服务能力）
- 网关统一对外提供入口，向 api-1 与 api-2 分发请求。

网关转发与切换配置位于：

- deploy/distributed/server1/gateway.conf

## 3. 场景一：关闭一台服务器，另一台继续工作

### 3.1 触发过程

当某一台服务器不可用（例如服务器 2 宕机或容器全部停止）时：

1. 网关访问该节点上游会失败。
2. 网关依据上游失败重试与切换策略，将请求转发到仍存活的另一节点。
3. 客户端入口地址不变，仍通过同一公网地址访问。

### 3.2 能继续“工作”的核心原因

不仅是 health 接口可访问，还需要核心业务接口可用。当前可持续工作的关键条件有三点：

1. 网关具备上游切换能力，避免单节点故障直接导致入口不可用。
2. API 认证采用无状态签名会话，切到另一实例时仍可验签，不依赖单进程内存会话。
3. 存活节点具备本机依赖能力（数据库/缓存/消息服务可本机提供），不会因远端依赖不可达而导致业务中断。

## 4. 场景二：重启一台服务器后，两台同时工作

### 4.1 恢复过程

当被重启节点恢复后：

1. 该节点 API 健康检查恢复通过。
2. 网关将其重新纳入可用上游池。
3. 新进入网关的请求会在两台实例之间继续分发。

### 4.2 如何判断“两台同时工作”

可通过以下证据确认：

1. 两个实例健康检查均返回 200。
2. 连续访问实例探针接口时，可统计到 api-1 与 api-2 均有命中。
3. 连续访问核心业务接口（例如 dashboard）时，响应头 x-instance-id 在两台实例间分布。

## 5. 关键技术点

1. 流量入口高可用
- 统一入口在 gateway，屏蔽后端节点变化。
- 通过 upstream + next_upstream 在节点失败时快速切换。

2. 会话跨实例可用
- 认证令牌为签名令牌，任一实例均可校验。
- 避免了单实例内存会话导致的跨实例登录失效。

3. 节点自治能力
- 每台业务节点具备可独立运行的必要依赖能力。
- 在单节点故障场景下，另一节点可维持核心链路。

## 6. 验收结论口径

满足以下条件可判定“支持分布式工作”：

1. 关闭任意一台服务器后，公网入口仍可访问。
2. 登录与至少一个核心业务接口可成功返回。
3. 故障节点恢复后，双节点都参与流量处理。
